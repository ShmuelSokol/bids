## Architecture locks

### PO write-back = Flow A confirmed
- DIBS generates TWO files: header DMF + lines DMF
- Header imported FIRST with the "auto-generate PO number" DMF checkbox turned ON — AX assigns PO numbers from its own UI number sequence
- DIBS then QUERIES AX via OData for the PO numbers
- Lines file generated with those PO numbers + line numbers → second DMF import
- Yosef will set up two DMF "projects" for Abe (one for header, one for lines) so Abe runs them without touching the mapping
- **DO NOT let Abe modify DMF mapping** — that auto-generate checkbox getting unchecked would break everything

### Sales orders — DO NOT build our own DMF generator
This was the biggest correction from the agenda. Yosef has a LOT of logic already built into AX's existing MPI sales order page. We feed it, we don't rebuild it.

**The current Abe flow:**
1. Abe goes to LamLinks → Awards → "Download since MM/DD" → Excel file lands on Desktop
2. VBA macro strips partial-day awards (only full days processed — today's partial day always excluded)
3. Abe uploads the stripped file to MPI Sales Order page in Dynamics (customer = DD219)
4. MPI shows errors (missing DODAAC map, missing NSN match)
5. Abe resolves errors in-page (creates items, maps DODAACs)
6. Click Import → batch creates SOs with correct header grouping, CLIN/TCN numbers, etc.

**Yosef's SO logic (do not reimplement):**
- Groups same contract # into one SO header
- UNLESS same contract has different DODAACs → splits into separate SOs
- DODAAC → mapped to Dynamics address_id via a lookup table (different DODAACs can point to the same physical address → same SO)
- Line gets CLIN + TCN
- Queries AX barcode table by NSN → returns ItemNumber + UoM (with multiple lookup via B3/B4/B5/B6 codes)

**Our job for DIBS (pre-validation layer, not replacement):**
- Auto-download the awards file from LamLinks (no desktop drop, no VBA needed)
- Pre-query AX:
  - For each DODAAC: is it mapped to an address_id? (If not, show it in DIBS with a prompt)
  - For each NSN: is there an item match with a valid UoM conversion?
- Show errors in DIBS BEFORE Abe goes to Dynamics
- Hand the cleaned file to Abe; he uploads to the same MPI Sales Order page unchanged
- MPI does the heavy lifting; we just cut the validation loop short

## Feature asks (new scope)

### A) Data source indicators — EVERY data point
Hover tooltip or badge on every number/text field showing where it came from: LamLinks / AX / MasterDB / DIBBS scrape / computed. Yosef was very specific about this.

### B) PO follow-up notifications
- Per-supplier SLA rules (McMaster = 1 day, Cardinal = same day, Med-Plus = 2 weeks, etc.)
- DIBS watches outstanding POs, surfaces overdue ones
- One-click "send follow-up email" using Dynamics vendor email addresses (not MDB — Dynamics is the source of truth for vendor email)
- This solves the "POs that never reached the vendor" problem Abe flagged costing real money

### C) Shipping consolidation
Action item for Yosef (speaks to his developer): add to the shipping software a feature where the operator scans all orders, sees destinations grouped, picks one destination, fills box info, prints label, and the SAME tracking number posts to every order on that shipment. That unlocks:
- Building proper pallets to single-depot destinations
- Weekly pallet shipments instead of per-order boxes
- Rules: real depots → consolidate; small front offices → don't (POD confusion)

### D) Mill Strip (WAWF tracking number passthrough)
Yosef / Abe get a lot of emails asking for POD. WAWF has a "Mill Strip" feature that accepts tracking number on the invoice — would eliminate most of those emails. **Open:** do we have EDI/API access to it? LamLinks has a tracking field on the invoice line but doesn't transmit to WAWF. Research item.

### E) Invoice post button automation (via SQL profiling)
Yosef confirmed with Maurice (his SQL guy) that we CAN set up SQL Profiler / a watch window to capture what happens when Abe clicks "Post" in LamLinks. Plan:
1. Abe opens LamLinks, gets ready to click Post on one invoice
2. Claude sets up SQL watch
3. Abe clicks Post
4. We capture the exact stored-proc call or SQL statement
5. DIBS replays it — "bulk post" / "post all ready invoices" becomes a one-click action in DIBS
6. Blocks hundreds of lost-payment cases where Abe saved invoices but never clicked Post

### F) Supplier email automation (deferred to Dynamics)
Do NOT duplicate vendor contact info in DIBS or MDB. Dynamics already has:
- Send Confirmation button (emails PDF to vendor using Dynamics-stored address)
- Already works, already automated
- Abe selects N POs, clicks Send Confirmation, all go out

We just need to make sure DIBS-created POs land in AX in a state where Send Confirmation works (which it will, since they're normal POs).
